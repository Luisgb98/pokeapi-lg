## The database decision: PostgreSQL, not MongoDB

**Decision (made by the advisor): PostgreSQL.** Rationale, specific to this
codebase — not a generic preference:

1. **The data is relational and constraint-heavy.** A user has many favorites
   (FK → pokemon id), many teams, each team has ≤6 ordered members, and (per
   plan 033) each member has an ability + nature + bounded IV/EV vectors + ≤4
   moves drawn from a legal learnset. These are exactly the join-and-constrain
   patterns relational databases enforce at the schema level
   (`CHECK`, `UNIQUE`, `FK`, `ON DELETE CASCADE`). In Mongo those invariants
   would have to be re-enforced in application code everywhere.
2. **The domain already thinks in typed, validated records.** Entities are
   readonly interfaces; boundaries are guarded with zod (plan 018). A typed SQL
   schema (Drizzle infers TS types from the schema) is a near-1:1 fit; the
   document model adds nothing here.
3. **Auth.js ships a first-class SQL adapter.** The users/accounts/sessions
   tables are a solved, documented schema.
4. **Serverless-friendly Postgres is the default in this ecosystem** (Neon /
   Vercel Postgres / Supabase) — HTTP-driver Postgres works from Next.js route
   handlers and server actions without connection-pool headaches.

MongoDB would only win if the saved structures were free-form, deeply nested,
schema-varying documents. They are not. **Do not revisit this; build on
Postgres.**

**Stack chosen:**

- **PostgreSQL** (provider: Neon or Vercel Postgres — operator picks; both are
  `postgres://` URLs, code is identical).
- **Drizzle ORM** (`drizzle-orm`, `drizzle-kit`) — TypeScript-first, thin,
  matches the "many small files / explicit, no magic" house style better than
  Prisma's codegen. Migrations are SQL files in the repo.
- **Auth.js v5** (`next-auth@beta` / `@auth/core`) with the **Drizzle adapter**
  (`@auth/drizzle-adapter`). Providers: start with **GitHub OAuth** (one
  provider is enough to ship; add Google later). This avoids storing passwords.

## Architecture — where everything lives (hexagonal rules from CLAUDE.md)

| Concern                                     | Layer / path                                                                | Notes                                          |
| ------------------------------------------- | --------------------------------------------------------------------------- | ---------------------------------------------- |
| Table definitions (Drizzle schema)          | `infrastructure/db/schema.ts`                                               | infra owns the DB                              |
| DB client / connection                      | `infrastructure/db/client.ts`                                               | reads `DATABASE_URL`                           |
| Migrations                                  | `infrastructure/db/migrations/*.sql`                                        | generated by drizzle-kit                       |
| Auth config (providers, adapter, callbacks) | `infrastructure/auth/config.ts` + `src/app/api/auth/[...nextauth]/route.ts` | infra + app route handler                      |
| Domain ports                                | `domain/ports/UserDataRepository.ts`                                        | interface only, zero deps                      |
| Domain entities                             | `domain/entities/` (e.g. `SavedTeam.ts`)                                    | readonly interfaces                            |
| Adapter impl                                | `infrastructure/db/DrizzleUserDataRepository.ts`                            | implements the port                            |
| Use cases                                   | `application/usecases/userData/*.ts`                                        | domain-only imports, **zero React**            |
| Server actions (mutations)                  | `application/actions/userData.ts`                                           | mirrors existing `application/actions/game.ts` |
| Container wiring                            | `application/container.ts`                                                  | add `getUserDataRepository()`                  |
| Session access for actions                  | `infrastructure/auth/session.ts` (`auth()` helper)                          | actions call it to get the user id             |
| Client state / sync                         | `presentation/store/*` + `presentation/hooks/*`                             | dual-mode (see Phase 4)                        |
| Auth UI                                     | `presentation/components/organisms/AuthMenu.tsx` etc.                       | sign-in/out, account menu                      |

**Hard rules to honor:** `application/` must contain **zero React imports**
(server actions are fine — they are not React). `presentation/` must **not**
import `infrastructure/` directly — it goes through server actions / the
container, exactly like today's pages do. Secrets (`DATABASE_URL`,
`AUTH_SECRET`, `AUTH_GITHUB_ID`, `AUTH_GITHUB_SECRET`) live in env only;
validate their presence at startup and never expose them to the client.

## Guest mode stays the default

The app must remain fully usable **logged out** — favorites/team/compare keep
working via the existing zustand+localStorage stores. The database is
**additive**: when a user signs in, their data syncs to/from the server. Do not
remove or gate the local stores behind auth. This keeps the SSR-first, no-login
experience intact and makes the DB an enhancement, not a wall.

## Phases

### Phase 0 — Decision, deps, env scaffolding

1. Add deps: `drizzle-orm`, the Postgres driver for the chosen host
   (`@neondatabase/serverless` for Neon, or `postgres` for node-postgres),
   `next-auth@beta`, `@auth/drizzle-adapter`. Dev: `drizzle-kit`.
2. Create `.env.example` (committed) documenting **names only**:
   `DATABASE_URL=postgres://...`, `AUTH_SECRET=` (generate with
   `npx auth secret`), `AUTH_GITHUB_ID=`, `AUTH_GITHUB_SECRET=`,
   `AUTH_URL=http://localhost:3000`. Real values go in `.env.local` (git-ignored
   — confirm `.gitignore` covers it). **Never commit real secrets.**
3. Add a startup validation module that throws a clear error if required env
   vars are missing in production (`infrastructure/db/env.ts` using zod).
4. Add `drizzle.config.ts` at repo root pointing at `infrastructure/db/schema.ts`
   and `infrastructure/db/migrations`.

**Gate**: `pnpm install` + `pnpm typecheck` exit 0; `.env.example` exists;
secrets are NOT committed (`git status`, grep for tokens).

### Phase 1 — Schema + migrations

Create `infrastructure/db/schema.ts` with Drizzle tables:

- **Auth.js adapter tables** (`users`, `accounts`, `sessions`,
  `verificationTokens`) — copy verbatim from the `@auth/drizzle-adapter`
  Postgres reference schema (do not hand-roll these).
- **`favorites`**: `userId` (FK → users, `ON DELETE CASCADE`),
  `pokemonId int`, `createdAt timestamptz default now()`. `PRIMARY KEY (userId,
pokemonId)` (a favorite is unique per user).
- **`teams`**: `id uuid pk`, `userId` (FK cascade), `name text not null`,
  `createdAt`, `updatedAt`.
- **`teamMembers`**: `id uuid pk`, `teamId` (FK → teams, cascade),
  `slot smallint not null` (0–5, `CHECK 0..5`), `pokemonId int not null`.
  `UNIQUE (teamId, slot)`. **Leave room for plan 033**: this is where the
  ability/nature/level/ivs/evs/moves columns will be added later — add a
  comment marking that, but do NOT add them now (033 owns that migration).
- **`comparisons`**: `id uuid pk`, `userId` (FK cascade), `name text`,
  `slotA int`, `slotB int`, `slotC int` (nullable), `createdAt`. Mirrors the
  three-slot `compareStore` shape.

Generate the migration (`pnpm db:generate`) and apply it (`pnpm db:migrate`)
against a real dev Postgres. Commit the generated SQL.

**Gate**: migration applies cleanly; `drizzle-kit` reports schema in sync;
tables exist (verify with `pnpm db:studio` or `psql \dt`).

### Phase 2 — Authentication

1. `infrastructure/auth/config.ts` — Auth.js config: Drizzle adapter wired to
   the client + schema, GitHub provider, `session: { strategy: 'database' }`,
   and a session callback that exposes `session.user.id`.
2. `src/app/api/auth/[...nextauth]/route.ts` — export the `GET`/`POST` handlers.
3. `infrastructure/auth/session.ts` — export the `auth()` helper (the
   server-side "who is logged in" accessor) for use in server actions and RSC.
4. `presentation/components/organisms/AuthMenu.tsx` (`'use client'`) — sign-in
   (GitHub) / sign-out / shows avatar + name when logged in. Wire it into
   `TopNav.tsx` (right cluster, beside `ThemeToggle`/`LanguageSwitcher`).
5. i18n: `auth` namespace (`signIn`, `signOut`, `account`, `guest`…) in all six
   `messages/*.json`.

**Gate**: `pnpm dev` → click sign-in → GitHub OAuth round-trips → a `users`
row and `sessions` row appear → avatar shows in nav → sign-out clears it.
`pnpm typecheck` exit 0.

### Phase 3 — Domain port + use cases + adapter

1. `domain/ports/UserDataRepository.ts` (interface only):
   `getFavorites(userId)`, `setFavorite(userId, pokemonId, on)`;
   `listTeams(userId)`, `getTeam(userId, teamId)`, `saveTeam(userId, team)`,
   `deleteTeam(userId, teamId)`; `listComparisons(userId)`,
   `saveComparison(userId, c)`, `deleteComparison(...)`. Returns domain
   entities (`SavedTeam`, `SavedComparison` — new readonly interfaces in
   `domain/entities/`).
2. `infrastructure/db/DrizzleUserDataRepository.ts` — implements the port using
   the Drizzle client. All writes scoped by `userId` (authorization: a user can
   only read/write their own rows — enforce in every query, never trust a
   client-supplied userId).
3. `application/container.ts` — add `getUserDataRepository()` factory
   (singleton, mirrors `getRepository()`).
4. Unit-test the use cases / repository with a test Postgres or an in-memory
   fake implementing the port (the port abstraction makes this clean). Keep
   ≥80% coverage on `domain/`+`application/`+`infrastructure/` per CLAUDE.md.

**Gate**: repository CRUD tests pass; `pnpm test:coverage` thresholds hold;
layer-boundary ESLint (plan 019) passes.

### Phase 4 — Server actions + dual-mode store sync

1. `application/actions/userData.ts` (`'use server'`) — thin actions:
   `toggleFavoriteAction`, `saveTeamAction`, `deleteTeamAction`,
   `saveComparisonAction`, etc. Each: call `auth()` for the userId (reject if
   unauthenticated), **zod-validate every argument** (untrusted client input,
   plan 018 rule), then delegate to `getUserDataRepository()`. Rate-limit
   mutations (a small per-user token bucket or the host's built-in limiter).
2. Wire the existing stores to **dual mode**:
   - **Logged out**: behave exactly as today (localStorage). No change.
   - **Logged in**: reads hydrate from the server (RSC/loader passes initial
     data, or a TanStack Query call), writes call the server action AND keep the
     optimistic local update (snapshot → apply → roll back on failure, per the
     web patterns rules). Prefer TanStack Query mutations (already a dep) for
     the server round-trip so caching/invalidation is consistent with the rest
     of the app.

   Keep this isolated behind a hook (e.g. `useFavoritesSync`) so the store
   internals barely change.

**Gate**: logged-in favorite/team/compare changes persist across reload and
across devices (verify via a second browser session). Logged-out behavior
unchanged. Optimistic rollback works (simulate a failing action).

### Phase 5 — One-time local→cloud import on first sign-in

On first authentication, detect existing localStorage favorites/team/compare
and offer a one-click "Import your local data to your account" (a dismissible
prompt). On confirm, push local state through the save actions, then mark
imported (a flag in localStorage) so it does not re-prompt. Never auto-merge
silently; never delete local data.

**Gate**: a guest who built a team, then signs in, is offered import; after
import the team is in the DB; declining leaves the DB empty and local intact.

### Phase 6 — Final hardening + docs

- Add a "Saved teams" / account area (list saved teams, load one into the
  builder, rename, delete). Minimal but real.
- Security pass (use the **security-reviewer** agent): authz on every query,
  no secret leakage, input validation, rate limits, CSRF (Auth.js handles its
  own; custom actions rely on same-origin server actions).
- Update `CLAUDE.md`: document the new `infrastructure/db` + `infrastructure/auth`
  layers, the `UserDataRepository` port, env vars, and the guest/auth dual-mode
  rule. Update `README.md` setup steps (DB provisioning, env, migrations).

**Gate**: `pnpm ci` exits 0; security-reviewer reports no CRITICAL/HIGH;
docs updated.

## Maintenance notes

- This plan deliberately ships **one** OAuth provider (GitHub). Adding Google
  is a config-only follow-up.
- `teamMembers` is schema-designed for plan 033 (a comment marks where the
  ability/nature/level/IV/EV/moves columns go) — but 033 owns that migration.
- Keep the local stores as the source of truth for guests forever; the DB is an
  enhancement, not a replacement. Any future "force login" idea is a separate
  product decision.
- Connection handling: prefer the host's serverless/HTTP driver to avoid
  exhausting Postgres connections from Next.js' many short-lived invocations.
